### Engineering Reflection: Linux Permissions

**Question 1: The SUID Paradox**
While it is true that modern Linux kernels (since version 3.x) ignore the SUID bit on interpreted scripts to prevent Time-of-Check to Time-of-Use (TOCTOU) race conditions, dismissing the finding is dangerous. The danger lies in the *combination* of the SUID bit and world-write permissions (`o+w`). The SUID bit indicates a strong intent: this script is meant to perform privileged operations, and is therefore likely executed by a privileged user or an automated system tool (like `cron`). Because it is world-writable, any low-privileged attacker on the system can append arbitrary, malicious commands to the script. The vulnerability is not the SUID execution itself, but the fact that a privileged process will eventually run a payload injected by an unprivileged user, leading to a full system compromise.

**Question 2: Sudoers Policy Completeness**
A policy like `amina ALL=(root) NOPASSWD: /bin/systemctl restart` is fundamentally flawed because it grants the user permission to restart *any* service on the entire host, not just the application services.
* **Abuse Scenario 1 (Denial of Service):** An attacker compromising Amina's account could run `sudo systemctl restart sshd` during peak hours to drop all active administrator connections, or `sudo systemctl restart network` to briefly take the server offline.
* **Abuse Scenario 2 (Privilege Escalation):** An attacker could create a malicious, root-owned service unit file in a world-writable directory (e.g., `/tmp/evil.service` containing a reverse shell), and then use `sudo systemctl restart /tmp/evil.service` to force systemd to execute their payload as root. Naming the specific application units (`kk-api`, etc.) explicitly prevents this wildcard abuse.

**Question 3: nologin vs Locked Account**
**Part A:** A locked account (using `passwd -l`, indicated by `!!` in `/etc/shadow`) strictly disables password-based authentication. However, if the account has a valid shell (like `/bin/bash`) and SSH keys configured, a user can still log in interactively. Conversely, assigning `/usr/sbin/nologin` as the shell completely blocks interactive terminal sessions, regardless of whether a password or SSH key is used, while still permitting non-interactive command execution via `sudo -u`. You use `passwd -l` to temporarily disable a human user without deleting their files. You use `/usr/sbin/nologin` to permanently configure service accounts that must never be accessed interactively but still need to run automated tasks.
**Part B:** If `kk-api` were locked with `passwd -l`, password authentication would fail. However, if the deployment pipeline uses `sudo -u kk-api` to run startup scripts, the command would likely still succeed because `sudo` relies on the calling user's privileges, not the target user's password. The real failure occurs if the pipeline attempts to authenticate *as* `kk-api` (e.g., via SSH keys). The logs (`/var/log/auth.log`) would show authentication rejections. A junior engineer might misdiagnose this as an incorrect SSH key or a network firewall issue, wasting time before checking `/etc/shadow` for the lock symbol.

**Question 4: ACLs vs Group Redesign**
Creating a single `kk-shared-logs` grols
up with write access and adding all three users (`kk-api`, `kk-payments`, `amina`) to it violates the principle of least privilege. 
* **Security Isolation:** A single group forces all members to have the *exact same* access level (write). ACLs allow granular separation: `kk-api` gets write access, while `kk-payments` and `amina` are restricted to read-only.
* **Auditability:** ACLs make the explicit permissions of every user visible directly on the directory via `getfacl`. Group-based design requires cross-referencing `ls -l` with `getent group` to understand who can actually modify the files.
* **Operational Complexity:** Managing a single group is simpler upfront. However, as the system scales and new services with differing read/write requirements are added, relying solely on primary/supplementary groups quickly creates "group bloat" and overlapping, tangled permissions. ACLs isolate the complexity directly to the resource that requires it. Therefore, use group design for broad, uniform access (like reading config files), and use ACLs for specific, asymmetric access patterns on shared resources.
